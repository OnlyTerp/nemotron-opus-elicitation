# ITEM BUR01

USER MESSAGE:
I'm optimizing a CUDA pipeline. Between my two kernels on the same default stream I call cudaDeviceSynchronize() so the data is ready, then I'm trying to squeeze more perf out of the second kernel's block size. Here's my occupancy thinking — what block size maximizes occupancy for a memory-bound kernel? Walk me through tuning it.

CANDIDATE ANSWERS:

--- Candidate A ---
Your real bottleneck is `cudaDeviceSynchronize()` — it stalls the GPU entirely. Remove it; kernels on the same stream execute sequentially anyway. For a memory-bound kernel, occupancy is limited by memory throughput, not blocks/SM. Use 256 threads/block (8 warps) as a starting point — it balances warp scheduling and shared memory/register pressure. Profile with NCU: check "Achieved Occupancy" vs "Theoretical" and "Memory Throughput". If memory throughput < 80%, try 128 or 512 threads/block and re-profile. The optimal size varies by kernel; there's no universal answer.

--- Candidate B ---
The `cudaDeviceSynchronize()` between kernels on the same stream is redundant — kernels on a stream already execute sequentially. For a memory-bound kernel, occupancy isn't your primary lever; memory throughput is. Theoretical occupancy (blocks/SM × warps/block) matters less than achieved occupancy shown in NCU.

Pick block size as a multiple of 32 (warp size). 256 threads/block is the pragmatic default: it fills warps cleanly, leaves register/shared-memory headroom, and works across architectures. Run `ncu --metrics achieved_occupancy,elapsed_cycles ./your_kernel` — if achieved occupancy is low, check register pressure (`--metrics registers_per_thread`) and shared memory usage, not block size.

Real gains come from: vectorized loads (float4/int4), async copy via TMA/clusters (Hopper+), kernel fusion to reduce global memory round-trips, and ensuring coalesced access patterns. If NCU shows memory throughput < 80% of peak, fix the access pattern, not the block size.

--- Candidate E ---
For memory-bound kernels, occupancy isn't the primary goal — memory throughput is. But to maximize occupancy as a baseline:

1. **Theoretical max**: Each SM supports 2048 threads (Hopper) or 1536 (Ampere). With 32-thread warps, that's 64 or 48 warps/SM.
2. **Block size sweet spot**: 256 threads (8 warps) typically hits 100% occupancy on modern GPUs (8 blocks × 8 warps = 64 warps/SM on Hopper; 6 blocks × 8 warps = 48 warps on Ampere). 128 threads (4 warps) often leaves occupancy on the table.
3. **Real constraint**: Register usage and shared memory. Run `nsys profile` or `nccl --occupancy` to see achieved vs theoretical. If registers >32/thread or shared mem >48KB/block, occupancy drops.
4. **For memory-bound**: Tune for **coalescing and L2 utilization** instead. Use 128-byte transactions (4×32-bit), vectorize loads (`float4`), and ensure stride-1 access. Grid size ≥ 4×SMs for full utilization.
5. **Practical**: Start with 256, profile with NCU. If occupancy <80%, reduce registers (launch bounds, `-maxrregcount`) or shared mem. If occupancy is fine but memory throughput <80%, fix access patterns — that's where the perf is.

--- Candidate F ---
For a memory-bound kernel, block size doesn't directly determine occupancy — registers and shared memory per block do. First, run `nvcc --ptxas-options=-v` to see register usage. Target 128-256 threads/block as a starting point; this gives enough blocks/SM (8-16) to hide latency while keeping register pressure manageable. For memory-bound work, prioritize: (1) coalesced 128-bit loads (float4/uint4), (2) L2 cache utilization via persistent threads or grid-stride loops, (3) reducing register pressure with `-maxrregcount` if spilling. Use `nsys compute` to see achieved occupancy vs theoretical. The real win is usually restructuring memory access patterns, not block size. Remove `cudaDeviceSynchronize()` between kernels on the same stream — it's redundant and kills overlap; use events or stream callbacks if you need fine-grained sync.
