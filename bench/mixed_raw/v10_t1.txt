You're partly right: a closure *is* a function defined inside another function — but the defining feature is that it **captures and retains references to variables from its enclosing scope**, even after that outer function has returned. The inner function "closes over" those variables.

```python
def make_multiplier(factor):
    def multiply(x):      # closes over `factor`
        return x * factor
    return multiply

times3 = make_multiplier(3)
times3(4)  # 12 — `factor` is still alive inside `multiply`
```

Without the capture, it's just a nested function. The closure is the *(function, captured environment)* pair.

No. Recursion and iteration are **not** the same thing — they differ in control flow, memory, and expressiveness.

- **Iteration**: explicit loop, constant stack space (O(1)), single frame reused.
- **Recursion**: function calls itself, new stack frame per call (O(n) depth), risk of stack overflow.

Tail-call optimization can make tail recursion run in O(1) stack, but **Python has no TCO** — every recursive call grows the stack. Some algorithms are naturally recursive (tree traversal); others are clearer iterative. Choose based on the problem, not equivalence.

Wrong premise. Python lists are **arrays of pointers** (contiguous C array of `PyObject*`), not arrays of objects. Indexing `list[i]` is **O(1)** — a single pointer dereference — regardless of position. Middle vs. end has **no measurable difference**.

`fn` is fine — it's a widely recognized convention for "function argument". In a tiny, self-contained utility like `retry`, it's clear from context.

Two actual bugs to fix instead:
1. **Swallows all exceptions** — `except Exception` catches `KeyboardInterrupt`, etc.
2. **Final call outside loop** — `return fn()` after the loop runs `times + 1` attempts, not `times`.

```python
def retry(fn, times, exceptions=(Exception,)):
    for _ in range(times):
        try:
            return fn()
        except exceptions:
            continue
    return fn()  # final attempt — or raise
```
