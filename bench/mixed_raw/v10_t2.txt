You're right that a closure is a nested function, but that's only half the mechanism. The defining property: it **captures and retains references to variables from its enclosing scope** — even after that outer function has returned.

```python
def make_adder(x):
    def adder(y):      # closure: captures `x`
        return x + y
    return adder

add5 = make_adder(5)
add5(3)  # 8 — `x` is still alive, bound to 5
```

The inner function alone isn't a closure until it *closes over* at least one free variable. So: nesting + capture = closure. You had the nesting.

They compute the same class of functions, but they are **not** "basically the same."

| Aspect | Iteration | Recursion |
|--------|-----------|-----------|
| Stack | O(1) | O(n) — each call adds a frame |
| Tail calls | N/A | Optimized only in some languages (not CPython) |
| Expressiveness | Loops + state | Natural for tree/graph, divide-and-conquer |

In CPython, recursion hits `RecursionError` ~1000 frames deep. Recursion isn't "a loop that calls itself" — it's a function that calls itself with a fresh stack frame each time. That frame cost is the fundamental difference.

False premise. Python lists are **arrays of pointers** (contiguous `PyObject*`), not contiguous objects. Indexing is **O(1) everywhere** — `list[i]` is pointer arithmetic + dereference, same cost at index 0, 5M, or 9,999,999.
