Close! A closure is a function *plus* the environment it captures from its enclosing scope. The inner function "closes over" variables from the outer function, keeping them alive even after the outer function returns.

```python
def make_adder(x):
    def adder(y):
        return x + y  # captures x
    return adder

add5 = make_adder(5)
add5(3)  # 8 — x=5 is still accessible
```

The key insight: `adder` doesn't just *live inside* `make_adder` — it *remembers* `x`. So: every closure is a nested function, but not every nested function is a closure (only if it actually references outer-scope variables).

They're equivalent in computational power, but not "basically the same." Recursion uses the call stack (each call = a frame; risk of stack overflow); iteration uses explicit loop state (constant stack). Recursion is often clearer for tree/divide-and-conquer; iteration for linear workflows. Python has no tail-call optimization, so deep recursion crashes where iteration wouldn't.

`fn` is fine — a common convention for callbacks. But your code has a bug: after the loop exhausts, the final `return fn()` runs one more unguarded attempt (times=3 → 4 attempts, raw exception propagates).

```python
def retry(fn, times):
    for i in range(times):
        try:
            return fn()
        except Exception:
            if i == times - 1:
                raise
```

def fib(n: int) -> int:
