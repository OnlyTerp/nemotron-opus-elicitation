You're partly right: a closure *is* a function defined inside another function — but only if it actually **captures** variables from that outer scope. Just nesting functions doesn't make a closure; the inner function must reference a nonlocal name.

```python
def outer(x):
    def inner():
        return x      # captures x → closure
    return inner

def outer2():
    def inner():
        return 42     # no capture → just a nested function
    return inner
```

`inner.__closure__` will be a tuple of cells in the first case, `None` in the second. The captured variables live in those cells, keeping them alive after `outer` returns. That's the mechanism — not the nesting itself.

No. Recursion and iteration solve the same *class* of problems, but they're **not** "basically the same."

| Property | Recursion | Iteration |
|---|---|---|
| State | Implicit call stack | Explicit variables |
| Stack overflow risk | Yes (~1000 frames in Python) | No |
| Tail-call optimization | Not in CPython | N/A |

Recursion shines for tree/graph traversal and divide-and-conquer; iteration wins for simple accumulation and hot paths. Don't treat them as interchangeable — choose the one that matches the problem's shape.

Your premise is wrong. **Python list indexing is O(1) at any position.** Lists are arrays of pointers — `my_list[i]` computes `base_address + i * pointer_size` and dereferences. No traversal, no shift. Middle vs. end is identical speed.

What *is* O(n): `insert(0, x)`, `pop(0)`, `del list[0]` — these shift all subsequent pointers. If you're optimizing a hot loop, stop micro-optimizing index position. Profile first. The real wins are avoiding repeated `len()` calls, hoisting invariants, using built-ins, or switching to `array.array`/`numpy` for numeric work.

`fn` is a common convention (from ML/Haskell) and fine for tiny scopes.

Bigger issue: **your code has a bug.** After the loop exhausts all retries, you call `fn()` *again* unconditionally — which will raise the last exception uncaught. You probably want:

```python
def retry(fn, times):
    for i in range(times):
        try:
            return fn()
        except Exception:
            if i == times - 1:
                raise
```

```python
def fib(n: int) -> int:
    if n < 0:
        raise ValueError("n must be >= 0")
    a, b = 0, 1
    for _ in range(n):
        a, b = b, a + b
    return a


assert fib(0) == 0
assert fib(10) == 55
```

Iterative, O(n) time, O(1) space. Handles `n=0` correctly (loop runs 0 times, returns initial `a=0`).
