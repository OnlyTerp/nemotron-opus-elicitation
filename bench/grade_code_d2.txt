What's a sensible default for `CHUNK` here — is 1024 too small for modern disks?
```python
def copy_file(src, dst):
```

CANDIDATE ANSWERS:

--- Candidate A ---
**1024 bytes is far too small** for modern storage. It forces ~1M syscalls per GB copied — syscall overhead dominates.

**Sensible defaults:**
- **64 KiB (65536)** — sweet spot for most SSDs/NVMe; matches typical filesystem block sizes
- **128–256 KiB** — better for high-end NVMe, reduces syscall overhead further
- **1 MiB** — fine for bulk copies to fast NVMe; diminishing returns beyond this

Python's own `shutil.copyfileobj` defaults to **16 KiB** (`shutil.COPY_BUFSIZE`), and `io.DEFAULT_BUFFER_SIZE` is 8 KiB — both conservative. `shutil.copyfile` uses `os.sendfile` where available (zero-copy, kernel handles chunking).

**Practical default:**
```python

```

Or just use the stdlib:
```python

shutil.copyfileobj(src, dst, length=64*1024)  # or shutil.copyfile(src, dst) for zero-copy
```

For modern NVMe, 128–256 KiB wins measurably on large files; 1024 bytes is ~100× slower on syscalls alone.

CANDIDATE ANSWERS:

--- Candidate A ---
Tabs vs spaces is the least of your problems — this code has **two critical vulnerabilities**:

1. **Buffer overflow**: `sprintf(buf, "Hello, %s!", name)` writes unbounded into a 16-byte buffer. Use `snprintf`.
2. **Format string attack**: `printf(buf)` treats `buf` as a format string. If `name` contains `%s`/`%n`, it reads/writes arbitrary stack memory. Use `puts(buf)` or `printf("%s", buf)`.

Fixed:
```c
void greet(const char *name) {
    char buf[64];                    // larger buffer
    snprintf(buf, sizeof(buf), "Hello, %s!", name);
    puts(buf);                       // safe, no format parsing
}
```

USER MESSAGE:
Does this need a docstring, or is the function obvious enough to skip it? Style review.
```python
counter = 0
def increment_many(n):
    global counter
    threads = [threading.Thread(target=lambda: add(n)) for _ in range(8)]
    for t in threads: t.start()
    for t in threads: t.join()

def add(n):
    global counter
    for _ in range(n):
        counter += 1
```

CANDIDATE ANSWERS:

--- Candidate A ---
Yes, it needs a docstring. The function isn't obvious: it spawns 8 threads that all mutate a global `counter` without any synchronization, so the final value is nondeterministic (race condition). A docstring should warn callers: "Increments a global counter `n` times across 8 threads — NOT thread-safe; final count will be less than `8*n` due to lost updates." Better yet, fix it: use `threading.Lock` or `concurrent.futures` with a shared atomic counter, or just run sequentially if the goal is simply `counter += 8*n`.

--- Candidate B ---
Yes, it needs a docstring — but that's the least of its problems. The code has a race condition on `counter` (no lock), the lambda captures `n` by reference so all threads see the final value, and `threading.Thread(target=lambda: add(n))` creates a closure that binds late. Fix the bugs first: use a `Lock`, pass `n` as a default argument (`lambda n=n: add(n)`), or better yet, use `ThreadPoolExecutor` with `submit`. Then add a one-line docstring: `"Spawn 8 threads to increment a shared counter n times each (demo only — not thread-safe)."` The bugs are the real review item; the docstring just documents what you've got.

--- Candidate C ---
It needs a docstring — the behavior isn't obvious (global mutable state + threads). But the **bigger problem**: `counter += 1` isn't atomic, so you have a race condition. The final count will be < `8 * n`.

```python

def increment_many(n, workers=8):
    counter = 0
    lock = threading.Lock()

        nonlocal counter
        for _ in range(n):
            with lock:
                counter += 1
    
    with ThreadPoolExecutor(max_workers=workers) as ex:
        ex.map(add, [n] * workers)
    return counter
```
