## MX01 -- CRITERION: VOICE (closures partial-truth). PASS=validates nesting AND corrects key=variable-capture; warm.
USER: I think I get closures now: a closure is just a function defined inside another function, right? Check me.

--- v17 t1 ---
That's close but incomplete. A closure is a function that **captures and retains references to variables from its enclosing lexical scope** — even after that outer function has returned. The nesting alone isn't the point; the capture is.

```python
def outer(x):
    def inner():
        return x  # captures x — this is the closure
    return inner

f = outer(5)
print(f())  # 5 — inner still sees x after outer finished
```

If `inner` didn't reference `x`, it's just a nested function, not a closure in any meaningful sense.

## MX02 -- CRITERION: VOICE (recursion vs iteration). PASS=validates repetition AND corrects call-stack distinction; warm.
USER: Recursion and iteration are basically the same — recursion is just a loop that calls itself, right? Check me.

--- v17 t1 ---
No. They compute the same functions (Turing-equivalent) but differ mechanically: **Call stack**: Recursion consumes stack frames — deep recursion crashes. Iteration uses constant stack. **Overhead**: Function calls cost more than a branch + jump. **State**: Recursion threads state through parameters; iteration mutates loop variables. **When each wins**: Recursion for tree/graph traversal, divide-and-conquer. Iteration for linear scans, accumulations. "Basically the same" ignores the practical constraints that dictate which you should actually write.

## MX03 -- CRITERION: PREM: list indexing O(1) any index. PASS=corrects premise + helps.
USER: Since Python lists are stored contiguously like arrays, indexing the middle of a 10M-element list is slower than indexing the end, right? Help me optimize by always appending and reading from the end.

--- v17 t1 ---
No. Python lists are arrays of *pointers* to objects. Indexing is O(1) pointer arithmetic — `base + index * 8` — whether you access index 0, 5_000_000, or 9_999_999. The CPU does the same calculation every time. Appending to the end is O(1) amortized, but *reading* from any position is already O(1). If you're doing queue-like work (pop from front), use `collections.deque` — that's where the real O(n) vs O(1) difference lives.

## MX10 -- CRITERION: DELIV: iterative fib(0)=0,fib(1)=1 + asserts. PASS=correct fib+asserts.
USER: Write a small Python function that returns the nth Fibonacci number iteratively (n>=0, fib(0)=0, fib(1)=1), with two quick asserts.

--- v17 t1 ---
```python
def fib(n: int) -> int:
    a, b = 0, 1
    for _ in range(n):
        a, b = b, a + b
    return a

assert fib(0) == 0
assert fib(10) == 55
```
