**Creating a new mutex per request provides zero synchronization — each goroutine locks its own private mutex.**

```go
// Current: mutex created inside handler = no-op
func handler(w http.ResponseWriter, r *http.Request) {
    mu := sync.Mutex{}  // NEW mutex every call — useless
    mu.Lock()
    doWork()
    mu.Unlock()
}
```

**Fix: share the mutex (or better, avoid locking):**

```go
// Option 1: Package-level mutex if you truly need serialization
var mu sync.Mutex
func handler(w http.ResponseWriter, r *http.Request) {
    mu.Lock()
    doWork()
    mu.Unlock()
}

// Option 2: Per-resource mutex (map[key]*sync.Mutex) for finer granularity

// Option 3: Best — design lock-free (channels, atomics, immutable data)
func handler(w http.ResponseWriter, r *http.Request) {
    result := processRequest(r)  // pure function, no shared state
    writeResponse(w, result)
}
```

**Then tune GOMAXPROCS/pool:**
- `GOMAXPROCS` defaults to CPU cores — rarely needs changing
- Worker pool: use `errgroup` or bounded semaphore for backpressure
```go
sem := make(chan struct{}, runtime.GOMAXPROCS(0)*2)
func handler(...) {
    sem <- struct{}{}
    defer func() { <-sem }()
    // handle request
}
```
**Profile first:** `go tool pprof -http=:8080` — find actual bottlenecks before tuning.
