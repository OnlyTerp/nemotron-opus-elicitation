**Sorting before grep is completely wasted work — grep doesn't benefit from sorted input, and sort is O(n log n) vs grep's O(n).**

```bash
# Current: sort (slow, disk-heavy) + grep (fast) = terrible
sort huge.log > sorted.log && grep ERROR sorted.log | wc -l

# Fix 1: Just grep directly — 10-50× faster
grep -c ERROR huge.log

# Fix 2: If you need parallel grep (for complex patterns), use GNU parallel or split
# Split file, grep in parallel, sum counts
split -n l/$(nproc) huge.log chunk_
parallel -j $(nproc) "grep -c ERROR {}" ::: chunk_* | awk '{sum+=$1} END{print sum}'

# Fix 3: For repeated queries, index once with ripgrep (rg) or build a DB
rg --index ERROR huge.log  # builds index, subsequent searches instant
```

**Key points:**
- `grep -c` counts matches directly, no `wc -l` needed
- `sort` is only useful if you need sorted output (e.g., `uniq -c`, joins)
- For 5GB, single-threaded `grep` saturates disk I/O — parallelizing adds overhead
- If nightly: consider `ripgrep` (rg) which is faster than grep, or load into SQLite/ClickHouse for repeated queries
