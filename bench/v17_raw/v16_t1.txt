It's exponential O(2^n) — unusable past n≈35. Also `fib(0)` returns 0 correctly but `fib(1)` returns 1, so base case is fine. Fix: memoize (`@lru_cache`) or iterative loop. Iterative:
```python
def fib(n):
    a, b = 0, 1
    for _ in range(n):
        a, b = b, a + b
    return a
```
O(n) time, O(1) space.

No. Big-O describes *asymptotic growth rate*, not absolute time. It drops constants and lower-order terms — two O(n) algorithms can differ 1000x in real seconds. It tells you how time *scales* as n grows, not what the clock reads. Use it to compare algorithms, not to predict latency.

No. Python lists are contiguous arrays of *pointers*. Indexing is O(1) — pointer arithmetic + dereference — same cost at any position. Cache effects exist but are negligible for pure indexing; the pointer array fits in L2/L3. What *is* slower: inserting/deleting in the middle (O(n) shift).

def running_max(nums: list[int]) -> int:
    if not nums:
        raise ValueError("empty")
    total = nums[0]
    for n in nums:
        if n > total:
            total = n
    return total
