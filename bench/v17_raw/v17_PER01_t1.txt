You're right. asyncio is for **I/O-bound** work (network, disk, timers). For **CPU-bound** work in Python, neither asyncio nor threading helps — the GIL serializes CPU execution in both.

For CPU-bound work you need:
- **multiprocessing** — separate processes, each with its own GIL
- **C extensions / NumPy / PyTorch / etc.** — release the GIL in native code
- **subinterpreters (3.12+)** — still maturing

Threading only helps when the bottleneck is *waiting* (I/O, locks, `time.sleep`). If your loop does pure Python number-crunching, adding threads makes it *slower* from context-switch overhead.

That's the classic naive recursive Fibonacci — correct but exponential time. For `n=40` it's ~1 second; `n=50` is ~2 minutes. The redundancy is the problem: `fib(5)` calls `fib(3)` twice, `fib(2)` three times, etc.

Fix options:
**Memoization** (simplest, keeps recursion): `@lru_cache(maxsize=None) def fib(n): ...`
**Iterative** (O(n) time, O(1) space): `a, b = 0, 1; for _ in range(n): a, b = b, a + b; return a`
**Matrix exponentiation** (O(log n)) if you need massive `n`.

For typical use, the iterative version is the right call — no imports, no stack overflow, linear time.

No. Big-O describes how runtime *scales* with input size, not absolute seconds. It drops constants and lower-order terms: an O(n) algorithm might take 100ns per element or 10ms per element — Big-O doesn't distinguish.

What it tells you: if you double the input, an O(n) algorithm roughly doubles in time; O(n²) quadruples; O(log n) barely budges. Use Big-O to compare algorithms *asymptotically* and pick the right approach for your problem size. For real latency numbers, you benchmark.
