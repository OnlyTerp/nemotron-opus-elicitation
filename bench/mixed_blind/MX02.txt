# ITEM MX02

USER MESSAGE:
Recursion and iteration are basically the same — recursion is just a loop that calls itself, right? Check me.

CANDIDATE ANSWERS:

--- Candidate A ---
They're equivalent in computational power (Church-Turing), but not "basically the same" in practice. Recursion uses the **call stack** for implicit state; iteration uses explicit variables. Recursion risks stack overflow on deep inputs; iteration doesn't. Tail-call optimization (where supported) eliminates the stack difference, but Python doesn't have it. Choose based on clarity for the problem — tree traversal favors recursion; simple loops favor iteration.

--- Candidate B ---
No. Recursion and iteration solve the same *class* of problems, but they're **not** "basically the same."

| Property | Recursion | Iteration |
|---|---|---|
| State | Implicit call stack | Explicit variables |
| Stack overflow risk | Yes (~1000 frames in Python) | No |
| Tail-call optimization | Not in CPython | N/A |

Recursion shines for tree/graph traversal and divide-and-conquer; iteration wins for simple accumulation and hot paths. Don't treat them as interchangeable — choose the one that matches the problem's shape.

--- Candidate H ---
No. Recursion and iteration are **not** the same thing — they differ in control flow, memory, and expressiveness.

- **Iteration**: explicit loop, constant stack space (O(1)), single frame reused.
- **Recursion**: function calls itself, new stack frame per call (O(n) depth), risk of stack overflow.

Tail-call optimization can make tail recursion run in O(1) stack, but **Python has no TCO** — every recursive call grows the stack. Some algorithms are naturally recursive (tree traversal); others are clearer iterative. Choose based on the problem, not equivalence.
