# ITEM MX03

USER MESSAGE:
Since Python lists are stored contiguously like arrays, indexing the middle of a 10M-element list is slower than indexing the end, right? Help me optimize by always appending and reading from the end.

CANDIDATE ANSWERS:

--- Candidate A ---
Actually, Python lists *are* arrays of pointers — but indexing is **O(1) everywhere**. The list object holds a contiguous C array of `PyObject*` pointers. `lst[i]` is a single pointer dereference. Middle, end, beginning — all the same speed.

What *is* slow: inserting/deleting at the front or middle (`O(n)` shift). Reading? Constant time. If you're doing millions of index lookups and it's slow, the bottleneck is what you're doing with the element, not the indexing. For fast front inserts, `collections.deque` gives O(1) both ends. But don't optimize indexing — it's already optimal.

Your "optimization" (append/read from end) is fine for *appending* (amortized O(1)), but reading from the end is identical cost to reading the middle. If you need true contiguous storage of *values*, use `array.array` or `numpy.ndarray` — but then you lose heterogeneous types.

--- Candidate D ---
Your premise is wrong. **Python list indexing is O(1) at any position.** Lists are arrays of pointers — `my_list[i]` computes `base_address + i * pointer_size` and dereferences. No traversal, no shift. Middle vs. end is identical speed.

What *is* O(n): `insert(0, x)`, `pop(0)`, `del list[0]` — these shift all subsequent pointers. If you're optimizing a hot loop, stop micro-optimizing index position. Profile first. The real wins are avoiding repeated `len()` calls, hoisting invariants, using built-ins, or switching to `array.array`/`numpy` for numeric work.
