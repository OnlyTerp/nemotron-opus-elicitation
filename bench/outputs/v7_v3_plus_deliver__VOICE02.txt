Incorrect. REST is an **architectural style** (Fielding, 2000), not a protocol. HTTP is a protocol. They're often used together but are distinct:

**REST constraints**: client-server, stateless, cacheable, uniform interface, layered system, code-on-demand (optional). The uniform interface requires: resource identification (URIs), manipulation via representations, self-descriptive messages, HATEOAS.

**HTTP methods** (GET/POST/PUT/DELETE/PATCH) map to CRUD *if* you follow REST, but:
- REST ≠ HTTP: you can do REST over CoAP, WebSockets, gRPC (with caveats)
- HTTP ≠ REST: most "REST APIs" are actually RPC-over-HTTP (ignoring HATEOAS, using verbs in URLs, tunneling everything through POST)

Key difference: REST centers on **resources** (nouns) and hypermedia-driven state transitions. RPC centers on **operations** (verbs). Fielding explicitly said: if you're not doing HATEOAS, you're not doing REST — you're doing HTTP RPC.

So: REST uses HTTP methods *semantically*, but the methods alone don't make it REST.
