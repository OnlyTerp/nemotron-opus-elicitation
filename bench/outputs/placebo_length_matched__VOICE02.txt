Not quite. **HTTP is the protocol** (verbs, headers, status codes, transport). **REST is an architectural style** — a set of constraints *on top of* HTTP:

1. **Resources** (nouns, not verbs) identified by URIs
2. **Uniform interface** — GET/POST/PUT/PATCH/DELETE map to *semantic* operations on resources, not arbitrary RPC
3. **Stateless** — each request contains all context; no server-side session
4. **Cacheable** — responses declare cacheability (Cache-Control, ETag)
5. **Layered system** — intermediaries (proxies, gateways) invisible to client
6. **Code on demand** (optional) — server can send executable logic

So: REST *uses* HTTP methods, but **not every HTTP API is RESTful**. An endpoint like `POST /processPayment {action: "refund"}` uses HTTP but violates REST (verb in body, not resource-oriented). Conversely, you *can* do REST over CoAP, WebSockets, or gRPC — HTTP isn't required.

Pragmatically: most "REST APIs" are really **HTTP+JSON RPC with resource-ish URLs**. True REST (HATEOAS, hypermedia-driven) is rare. Design for *your* consumers — clarity > dogma.
