**This approach cannot work.** TCP provides reliable *in-order* delivery, not exactly-once. Disabling retries turns at-least-once into at-most-once (messages *will* be lost on any transient failure).

**Why TCP ≠ exactly-once:**
- TCP ACKs only mean "bytes received by kernel," not "processed by application"
- App crashes after ACK but before processing = lost message
- Network partition during processing = duplicate on retry (or loss without retry)

**What actually works for exactly-once:**

1. **Idempotent consumers** (practical standard): Design processing to be safely retryable. Use deduplication keys (message ID + consumer ID) stored in a transactional outbox table.

2. **Transactional outbox pattern**: Write message + business state in same DB transaction. Separate publisher reads outbox and sends (with retries). Consumer deduplicates via processed-message table.

3. **Kafka/RabbitMQ transactions**: Producer sends in transaction; consumer processes + commits offset in same transaction (Kafka exactly-once semantics).

**Socket options you asked for (but won't solve this):**
```c
// These just tune TCP reliability, not delivery semantics
setsockopt(fd, SOL_SOCKET, SO_KEEPALIVE, &on, sizeof(on));
setsockopt(fd, IPPROTO_TCP, TCP_USER_TIMEOUT, &ms, sizeof(ms));  // fail fast on partition
// No option for "exactly once"
```

**Bottom line:** Exactly-once is an application-layer concern. Build idempotency or use a message broker with transactional semantics. Socket options are irrelevant.
