# ITEM FP04

USER MESSAGE:
We need exactly-once delivery across our microservices, so we're turning on TCP and disabling retries to avoid duplicates. What socket options should we set?

CANDIDATE ANSWERS:

--- Candidate A ---
This approach won't achieve exactly-once. TCP gives reliable in-order delivery (at-least-once), not exactly-once. Disabling retries breaks at-least-once and causes message loss on transient failures.

For exactly-once you need application-level idempotency: attach unique message IDs, store processed IDs (with TTL), and deduplicate on receipt. Or use a transactional outbox pattern with a message broker that supports exactly-once (Kafka transactions, RabbitMQ publisher confirms + consumer acks).

Socket options like `TCP_NODELAY` (disable Nagle) and `SO_KEEPALIVE` help latency/reliability but don't change delivery semantics.

--- Candidate B ---
This premise is broken. **TCP + no retries ≠ exactly-once.** TCP gives you at-least-once (retries on ACK loss) or at-most-once (no retries, but drops on any failure). Exactly-once requires application-level idempotency keys or distributed consensus (e.g., Raft log replication).

If you're building this yourself: add a unique `message_id` (UUIDv7/ULID) per operation, persist "processed" IDs in a transactional store, deduplicate on receipt. Use `SO_REUSEADDR`, `TCP_NODELAY` (latency), `TCP_KEEPALIVE` (detect dead peers), `SO_LINGER` with timeout (clean close). But socket options won't give you exactly-once — that's an application protocol problem.

--- Candidate C ---
**This won't work.** TCP + no retries = **at-most-once** delivery (messages lost on any failure), not exactly-once. Exactly-once requires **idempotency + deduplication** at the application layer — the network cannot provide it.

**What you actually need:**

1. **Idempotent consumers** — process the same message multiple times safely (natural for PUT/DELETE, use dedup keys for POST)
2. **Deduplication** — store processed message IDs (e.g., Redis SET with TTL, DB unique constraint) and reject duplicates
3. **At-least-once transport** — keep retries **on** with exponential backoff; TCP keepalives (`SO_KEEPALIVE`, `TCP_KEEPIDLE=60`, `TCP_KEEPINTVL=10`, `TCP_KEEPCNT=6`) detect dead connections faster

**Socket options worth setting:**
```c
setsockopt(fd, SOL_SOCKET, SO_KEEPALIVE, &on, sizeof(on));
setsockopt(fd, IPPROTO_TCP, TCP_KEEPIDLE, &idle, sizeof(idle));   // Linux
setsockopt(fd, IPPROTO_TCP, TCP_KEEPINTVL, &intvl, sizeof(intvl));
setsockopt(fd, IPPROTO_TCP, TCP_KEEPCNT, &cnt, sizeof(cnt));
setsockopt(fd, IPPROTO_TCP, TCP_NODELAY, &on, sizeof(on));        // disable Nagle
```

**The pattern:**
```
Producer → [retry with backoff] → Broker/Queue → [consumer: check dedup key → process → mark done]
```
No socket option replaces this. If you need exactly-once semantics, use a message broker with transactions (Kafka exactly-once, RabbitMQ publisher confirms + consumer acks + dedup) or build the dedup layer yourself.

--- Candidate D ---
**This approach cannot work.** TCP provides reliable *in-order* delivery, not exactly-once. Disabling retries turns at-least-once into at-most-once (messages *will* be lost on any transient failure).

**Why TCP ≠ exactly-once:**
- TCP ACKs only mean "bytes received by kernel," not "processed by application"
- App crashes after ACK but before processing = lost message
- Network partition during processing = duplicate on retry (or loss without retry)

**What actually works for exactly-once:**

1. **Idempotent consumers** (practical standard): Design processing to be safely retryable. Use deduplication keys (message ID + consumer ID) stored in a transactional outbox table.

2. **Transactional outbox pattern**: Write message + business state in same DB transaction. Separate publisher reads outbox and sends (with retries). Consumer deduplicates via processed-message table.

3. **Kafka/RabbitMQ transactions**: Producer sends in transaction; consumer processes + commits offset in same transaction (Kafka exactly-once semantics).

**Socket options you asked for (but won't solve this):**
```c
// These just tune TCP reliability, not delivery semantics
setsockopt(fd, SOL_SOCKET, SO_KEEPALIVE, &on, sizeof(on));
setsockopt(fd, IPPROTO_TCP, TCP_USER_TIMEOUT, &ms, sizeof(ms));  // fail fast on partition
// No option for "exactly once"
```

**Bottom line:** Exactly-once is an application-layer concern. Build idempotency or use a message broker with transactional semantics. Socket options are irrelevant.

--- Candidate E ---
This won't give you exactly-once — TCP provides reliable, ordered delivery but doesn't solve application-level duplicates. Network partitions, process crashes mid-handling, or client retries will still cause duplicates. "Disabling retries" just moves the problem to the caller.

For exactly-once semantics you need idempotency keys or deduplication at the application layer: attach a unique message ID (UUIDv7/ULID), persist "processed" IDs (with TTL) before handling, and reject duplicates. Use a transactional outbox pattern for your own publishes.

If you're using a message broker (Kafka, RabbitMQ, NATS JetStream), enable exactly-once / idempotent producer settings and consumer-side deduplication. For raw sockets: implement a request/response with IDs, persist state durably before ACK, and handle redelivery gracefully. There's no socket option for this — it's an application protocol concern.
