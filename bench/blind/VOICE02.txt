USER MESSAGE:
REST and HTTP are basically the same thing — REST is just the HTTP methods GET/POST/PUT/DELETE, correct? Check me.

CANDIDATE ANSWERS:

--- Candidate A ---
**Incorrect.** REST and HTTP are fundamentally different things:

| Aspect | HTTP | REST |
|--------|------|------|
| **What it is** | A **protocol** (transport) | An **architectural style** (constraints) |
| **Scope** | Wire format: verbs, headers, status codes, body | Design principles for distributed systems |
| **Relationship** | REST *commonly uses* HTTP | REST *can use* other protocols (CoAP, gRPC, WebSockets) |

**REST constraints (Fielding, 2000):**
1. Client-server separation
2. Statelessness
3. Caching
4. Uniform interface (resource identification, manipulation via representations, self-descriptive messages, HATEOAS)
5. Layered system
6. Code on demand (optional)

**HTTP methods ≠ REST.** You can:
- Use HTTP *without* REST (RPC over HTTP, SOAP, GraphQL)
- Use REST *without* HTTP (e.g., REST over CoAP for IoT)

**What people often mean**: "RESTful HTTP API" — an API that maps resources to URLs, uses HTTP verbs semantically (GET=read, POST=create, PUT/PATCH=update, DELETE=remove), returns appropriate status codes, and includes hypermedia links (HATEOAS). Most "REST APIs" are actually **HTTP JSON RPC** — they violate HATEOAS and uniform interface.</think>

No. REST is an **architectural style** (Representational State Transfer), not a protocol. HTTP is a transport protocol. They're orthogonal.

REST constraints (Fielding 2000): client-server, stateless, cacheable, uniform interface (resources identified by URIs, manipulated via representations, self-descriptive messages, HATEOAS), layered system, code-on-demand optional.

HTTP methods map *conventionally* to CRUD (GET/POST/PUT/PATCH/DELETE), but you can do REST over WebSockets, gRPC, or carrier pigeon. Conversely, you can use HTTP with RPC semantics (POST /doAction) — that's not REST.

What makes it REST: resources have stable URIs, state transitions driven by hypermedia (HATEOAS), uniform interface. Most "REST APIs" are actually HTTP+JSON RPC — they skip HATEOAS and couple clients to hardcoded endpoints.

--- Candidate D ---
Not quite. **HTTP is the protocol** (verbs, headers, status codes, transport). **REST is an architectural style** — a set of constraints *on top of* HTTP:

1. **Resources** (nouns, not verbs) identified by URIs
2. **Uniform interface** — GET/POST/PUT/PATCH/DELETE map to *semantic* operations on resources, not arbitrary RPC
3. **Stateless** — each request contains all context; no server-side session
4. **Cacheable** — responses declare cacheability (Cache-Control, ETag)
5. **Layered system** — intermediaries (proxies, gateways) invisible to client
6. **Code on demand** (optional) — server can send executable logic

So: REST *uses* HTTP methods, but **not every HTTP API is RESTful**. An endpoint like `POST /processPayment {action: "refund"}` uses HTTP but violates REST (verb in body, not resource-oriented). Conversely, you *can* do REST over CoAP, WebSockets, or gRPC — HTTP isn't required.

Pragmatically: most "REST APIs" are really **HTTP+JSON RPC with resource-ish URLs**. True REST (HATEOAS, hypermedia-driven) is rare. Design for *your* consumers — clarity > dogma.

--- Candidate E ---
**No — that's a category error.** REST is an *architectural style* (constraints: client-server, stateless, cacheable, uniform interface, layered system, code-on-demand). HTTP is a *protocol* that happens to implement the uniform interface well.

**What you described (mapping CRUD to GET/POST/PUT/DELETE) is just "HTTP API"** — often called "RESTful" but missing most constraints.

**True REST requires:**
- **Hypermedia as engine of state (HATEOAS):** responses include links/relations so clients *discover* actions, not hardcode URLs
- **Self-descriptive messages:** media types (e.g., `application/hal+json`) define semantics, not out-of-band docs
- **Statelessness:** zero server-side session state; auth via tokens in headers
- **Cacheability:** explicit `Cache-Control`, `ETag`, `Last-Modified` on every response

**Most "REST APIs" are actually RPC-over-HTTP** — fixed endpoints, JSON payloads, no hypermedia. That's fine for many cases, but don't call it REST if you violate the constraints.

**Litmus test:** Can a generic client (like a browser) navigate your API solely by following links in responses, with zero prior knowledge of your URL structure? If no, it's not REST.
