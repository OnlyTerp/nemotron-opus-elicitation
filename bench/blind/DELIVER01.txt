# ITEM DELIVER01

USER MESSAGE:
Our React list re-renders the whole page on every keystroke, but we read that useMemo fixes all re-render problems, so wrap the component in useMemo for me and that should fix it.

CANDIDATE ANSWERS:

--- Candidate A ---
`useMemo` memoizes a *value*, not a component. Wrapping a component in `useMemo` does nothing — it returns the same component function. You likely need `React.memo` on the child list component, or memoize the *derived data* (e.g., `useMemo(() => list.filter(...), [list, filter])`) and pass stable callbacks via `useCallback`. Also check: are you passing new object/array refs as props each render? Fix the root cause (unstable refs), don't just wrap things in hooks.

--- Candidate B ---
Premise check: `useMemo` memoizes a *value*, not a component. Wrapping a component in `useMemo` is an anti-pattern — it returns the rendered output once, but the component still re-executes on every parent render.

What you likely want:
- `React.memo(Component)` — prevents re-render when props are referentially equal
- `useCallback` for handler functions passed as props
- `useMemo` for expensive *calculations* inside the component

Fix the actual cause: move list rendering to a child component wrapped in `React.memo`, stabilize props with `useCallback`/`useMemo`, or virtualize with `react-window`. `useMemo` around the whole component won't help.

--- Candidate C ---
Premise check: `useMemo` memoizes a *value*, not a component. Wrapping a component in `useMemo` does nothing — it returns the same component function, so React still re-renders it on every parent render.

What actually helps list re-renders on keystroke:
1. **Virtualize** the list (`react-window`, `@tanstack/virtual`) — only render visible rows.
2. **Memoize row components** with `React.memo(Row)` + stable keys.
3. **Debounce/throttle** the input state, or lift state up so the list doesn't re-render on every keystroke.
4. If rows are expensive, `useMemo` the *derived data* (filtered/sorted array), not the component.

`useMemo`/`useCallback` are for stabilizing referential identity passed to memoized children — not a blanket fix.

--- Candidate D ---
`useMemo` does **not** fix re-renders caused by a parent re-rendering — it only memoizes a *computed value* (or callback) *inside* a component. Wrapping the whole component in `useMemo` is a category error; that's what `React.memo` is for.

Your real problem: the list re-renders on every keystroke because (a) the parent re-renders on input change, or (b) the list items aren't memoized. Fixes:

1. **`React.memo` on the list item component** — prevents re-render when props are referentially equal.
2. **Stabilize callbacks** with `useCallback` in the parent so item `onClick`/`onChange` don't get new functions each render.
3. **Virtualize** (react-window) if the list is large — only render visible rows.
4. **Move state down** — keep the input state in a sibling or child, not in a common ancestor of the list.

`useMemo` helps when you have an expensive calculation *inside* a render: `const sorted = useMemo(() => expensiveSort(items), [items])`. It doesn't stop the component from re-running.

--- Candidate E ---
`useMemo` memoizes a *value*, not a component. Wrapping the component in `useMemo` does nothing — it still re-renders on every parent render. Your keystroke problem is likely: (1) the list component isn't memoized with `React.memo`, (2) the inline handler or `onChange` creates new functions each render, (3) the list items lack stable `key`s, or (4) state lives too high. Fix: `const List = React.memo(({ items, onChange }) => ...)` + `const handleChange = useCallback((e) => setFilter(e.target.value), [])` in the parent + stable keys. Profile with React DevTools "Highlight updates" first — `useMemo`/`useCallback` are optimizations, not fixes for architectural re-render causes.
