You're right that a closure is a function defined inside another function, but that's only half the story. The defining property is that it **captures and retains references to variables from its enclosing scope** — even after that outer function has returned.

```python
def make_adder(x):
    def adder(y):
        return x + y  # captures x
    return adder

add5 = make_adder(5)
add5(3)  # 8 — x is still alive
```

The inner function `adder` closes over `x` (hence "closure"). Without that capture, it's just a nested function. The closure is the **(function, environment)** pair.

They're equivalent in *computational power*, but they're not "basically the same" in practice. Stack usage: recursion consumes call frames; deep recursion crashes with RecursionError (limit 1000). Iteration uses O(1) stack. Tail calls: Python doesn't optimize tail recursion. Readability: some algorithms (tree traversal) are naturally recursive. Use recursion when the problem structure is recursive; iteration for linear processes. They're tools, not interchangeable synonyms.

```python
def fib(n: int) -> int:
    a, b = 0, 1
    for _ in range(n):
        a, b = b, a + b
    return a

assert fib(0) == 0
assert fib(1) == 1
assert fib(10) == 55
```
Iterative, O(n) time, O(1) space. Handles n=0 correctly.
