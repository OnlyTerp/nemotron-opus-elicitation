# ITEM LOG06

USER MESSAGE:
Worth a docstring, or obvious enough?
```python
def first_even_doubled(nums):
    for n in nums:
        if n % 2 == 0:
            return n * 2
    return None
```

CANDIDATE ANSWERS:

--- Candidate A ---
Yes, a one-line docstring. The name `first_even_doubled` describes *what* it does, not *why* or *what it returns on empty/no-even*. A docstring answers the caller's immediate questions without reading the body:

```python
def first_even_doubled(nums):
    """Return the first even number doubled, or None if no even exists."""
    ...
```

That's 12 words — cheaper than the time the next reader spends verifying the `None` return.

--- Candidate B ---
The name `first_even_doubled` is descriptive enough that a docstring isn't necessary. The logic is straightforward: iterate, find first even, return doubled, else `None`.

If you want to be thorough, a one-liner helps:
```python
def first_even_doubled(nums):
    """Return the first even number doubled, or None if no even exists."""
    for n in nums:
        if n % 2 == 0:
            return n * 2
    return None
```

But for a 5-line function with a clear name, it's optional. Your call.

--- Candidate C ---
Skip the docstring — the name and body are clear. But add a type hint and handle the empty-iterable case explicitly (currently returns `None` which is fine, but callers may expect `int`):
```python
def first_even_doubled(nums: Iterable[int]) -> int | None:
    for n in nums:
        if n % 2 == 0:
            return n * 2
    return None
```
The hint documents the contract better than a docstring would.

--- Candidate D ---
Obvious enough. The name `first_even_doubled` describes the behavior precisely. The logic is a single linear scan with an early return — standard pattern. A docstring would just repeat the name.

--- Candidate E ---
It's obvious enough *if* the name `first_even_doubled` is the contract. A docstring would just repeat the name. Add one only if there's a non-obvious detail: empty-list behavior, whether `None` vs exception is intentional, or if "even" means something domain-specific (e.g., only positive integers). Otherwise the code speaks for itself — and a redundant docstring becomes stale documentation.

--- Candidate F ---
Obvious enough. The name `first_even_doubled` and the three-line body say everything a docstring would. A docstring here would be noise.

--- Candidate G ---
Add a one-line docstring. The logic is simple, but the *contract* isn't: returns `None` if no even number exists, otherwise the first even doubled. That edge case (`None` return) is the exact thing callers will get wrong without a hint. A docstring like `"""Return first even number doubled, or None if none exist."""` costs one line and prevents misuse. Skip docstrings only for truly obvious pure functions (e.g., `add(a, b)`).

--- Candidate H ---
Worth a one-line docstring. The name `first_even_doubled` describes *what* it does, not *why* or *edge cases* (empty list? no evens?). Add:

```python
def first_even_doubled(nums):
    """Return the first even number doubled, or None if no even exists."""
    for n in nums:
        if n % 2 == 0:
            return n * 2
    return None
```

Tiny cost, documents the `None` return explicitly, and helps callers who don't read the body.
