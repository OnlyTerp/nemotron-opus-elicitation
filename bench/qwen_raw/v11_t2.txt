You're partly right: a function defined inside another is necessary, but that alone doesn't make a closure. The defining mechanic is that the inner function captures *free variables* from the enclosing scope — it holds live references to them, not snapshots. Even after the outer function returns and its stack frame is gone, the closure keeps those variables alive.

```python
def make_adder(n):
    def add(x):        # `n` is a free variable here
        return x + n    # captured by reference, not value
    return add
```

Key detail: Python binds by *name*, not by position. The variable `n` is looked up at call time. If the inner function doesn't reference any outer variable, it's just an inner function, not a closure.

"Recursion is just a loop that calls itself" misses the structural difference. Recursion and iteration are *computationally equivalent* (Church–Turing), but they're not the same mechanism. A loop mutates state in-place and re-enters the same stack frame. Recursion creates a *new stack frame* per step — each call gets its own bindings. **Termination:** a loop stops when the condition is false; recursion stops when a base case is hit. **Practical difference:** CPython has no TCO. A recursive factorial blows the stack at ~1,000 calls; a `while` loop won't. Some problems (tree traversal, backtracking) map cleanly to recursion. Pick based on stack depth and mental model, not equivalence.

That's wrong. Python lists are arrays of pointers, and indexing is O(1) at *any* position — the middle of a 10M-element list takes exactly the same time as indexing position 0 or 9,999,999. The "append and read from the end" advice is backwards. Appending to a list is amortized O(1) but inserting at the front is O(n) — and iterating forward from the end is just as fast. If you truly need constant-time appends/deletes from both ends, use `collections.deque`. But for indexed access, a list is already optimal regardless of position.

The JSON library is a red herring. Your actual bottleneck is two operations inside the hot loop that belong outside:

1. `len(my_list)` is O(1) and free in CPython (it reads `ob_size`). Don't optimize this.
2. `set(my_list)` rebuilds the entire set every iteration — O(n) work repeated n times, making your loop O(n²). Hoist it:

```python
lookup = set(my_list)  # O(n), done once
for item in my_list:
    if item in lookup:  # O(1) amortized
        ...
```

That alone will dwarf any JSON library savings. For serialization, `orjson` is typically 3–10× faster. But profile first.

Spaces. PEP 8 mandates 4 spaces. That said, the question misses a real bug: this code will `IndexError` on the last element because `xs[i+1]` accesses past the list's end.

```python
def is_sorted(xs):
    for i in range(len(xs) - 1):  # -1: don't access i+1 on last element
        if xs[i] > xs[i + 1]:
            return False
    return True
```

With that fix, the function correctly checks sorted order in O(n). But honestly, `all(xs[i] <= xs[i + 1] for i in range(len(xs) - 1))` is cleaner. And `xs == sorted(xs)` is the simplest read.
