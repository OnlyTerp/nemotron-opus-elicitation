You're close but your definition is backwards. A closure isn't "a function defined inside another function"—that's just a nested function. What makes it a closure is that the inner function **captures and retains references to variables from the outer scope**, even after the outer function has returned.

```python
def make_adder(n):
    def add(x):          # nested function
        return x + n     # captures `n` from outer scope
    return add
```

`add` is a closure because it carries `n` with it. The mechanism: Python stores captured variables in the closure's `__closure__` attribute. Your mental model needs one fix: the key property is **state retention across scope boundaries**, not nesting. Every nested function that references outer variables is a closure; every nested function that doesn't is just nested.

They share a surface similarity—both express repetition—but they solve fundamentally different problems. What you got right: recursion *does* call a function from within itself. Where it breaks: calling recursion "just a loop that calls itself" misses the critical distinction. **Iteration** advances state through an explicit control variable and runs in O(1) stack depth. **Recursion** creates a new stack frame per call, so its space complexity is O(n) without tail-call optimization (which Python explicitly doesn't provide). Iteration is for known, bounded repetition. Recursion is for self-similar, hierarchical structure (tree traversal, divide-and-conquer). Don't think "which is faster." Think "which matches the problem structure."

Your premise is wrong and your recommended optimization would make things worse. Python lists **are** arrays of pointers (PyObject*), and indexing is O(1)—it's a single pointer dereference at `base_address + index * 8`. There is no distance penalty. Indexing `lst[0]`, `lst[5000000]`, or `lst[9999999]` takes exactly the same time. Your suggestion to "always append and read from the end" would actually hurt you: you'd lose random access entirely. If you need O(1) amortized appends and O(1) indexing, `list` is already optimal. If you need frequent insertions at arbitrary positions, use `collections.deque` for O(1) ends.

Tabs or spaces isn't the real issue—your function has an `IndexError` bug. `range(len(xs))` produces indices `0` through `len(xs)-1`. When `i` reaches `len(xs)-1`, the line `xs[i] > xs[i+1]` tries to access `xs[len(xs)]`, out of bounds.

```python
def is_sorted(xs):
    for i in range(len(xs) - 1):    # stop one early
        if xs[i] > xs[i+1]:
            return False
    return True
```

But just use `all()`: `return all(xs[i] <= xs[i+1] for i in range(len(xs) - 1))`. For indentation: PEP 8 mandates **4 spaces**.
