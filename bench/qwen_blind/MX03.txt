# ITEM MX03

USER MESSAGE:
Since Python lists are stored contiguously like arrays, indexing the middle of a 10M-element list is slower than indexing the end, right? Help me optimize by always appending and reading from the end.

CANDIDATE ANSWERS:

--- Candidate A ---
Two misconceptions here. Python lists are arrays of *pointers* to objects, not contiguous numeric data. Indexing is O(1) regardless of position — accessing the middle is just as fast as the end. So "always read from the end" won't optimize anything. For appends, regular lists already support O(1) amortized append. Use `collections.deque` if you need O(1) pops from the left. Don't rewrite your access patterns based on incorrect assumptions about list internals.

--- Candidate B ---
That's wrong. Python lists are arrays of pointers, and indexing is O(1) at *any* position — the middle of a 10M-element list takes exactly the same time as indexing position 0 or 9,999,999. The "append and read from the end" advice is backwards. Appending to a list is amortized O(1) but inserting at the front is O(n) — and iterating forward from the end is just as fast. If you truly need constant-time appends/deletes from both ends, use `collections.deque`. But for indexed access, a list is already optimal regardless of position.

--- Candidate C ---
That's incorrect on two counts. First, Python list indexing is O(1) regardless of position. Lists are dynamic arrays of object pointers—Python computes the address as `base + index * pointer_size`. The middle index is no slower than the last. Second, the advice to "always append and read from the end" abandons random access entirely. For LIFO access, `list` with `.append()`/`.pop()` is O(1). For queues, `collections.deque`. Rebuilding a set every iteration is wasteful, but that's unrelated to indexing. Don't avoid middle indexing out of fear of slowdown that doesn't exist.

--- Candidate D ---
Your premise is wrong and your recommended optimization would make things worse. Python lists **are** arrays of pointers (PyObject*), and indexing is O(1)—it's a single pointer dereference at `base_address + index * 8`. There is no distance penalty. Indexing `lst[0]`, `lst[5000000]`, or `lst[9999999]` takes exactly the same time. Your suggestion to "always append and read from the end" would actually hurt you: you'd lose random access entirely. If you need O(1) amortized appends and O(1) indexing, `list` is already optimal. If you need frequent insertions at arbitrary positions, use `collections.deque` for O(1) ends.
